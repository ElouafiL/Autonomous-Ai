[💡 المبتكر]
REST (Representational State of Resource) و GraphQL هما نمطان مختلفان لتصميم واجهات برمجة التطبيقات (APIs). فيما يلي شرح للفرق بينهما مع مثال عملي لكل منهما، ومتى نستخدم كل تقنية:

**REST:**

REST هو نمط معماري شائع لتصميم واجهات برمجة التطبيقات. يعتمد على فكرة أن كل مورد (resource) يُمثَّل بواسطة معرف فريد (URI) ويتم الوصول إليه عبر طرق HTTP (GET، POST، PUT، DELETE).

مثال عملي على REST:

*   إذا كنت تريد الحصول على قائمة المستخدمين، قد ترسل طلب GET إلى `/users`.
*   إذا كنت تريد إنشاء مستخدم جديد، قد ترسل طلب POST إلى `/users` مع البيانات اللازمة.
*   إذا كنت تريد تحديث معلومات مستخدم، قد ترسل طلب PUT إلى `/users/{id}` مع البيانات الجديدة.
*   إذا كنت تريد حذف مستخدم، قد ترسل طلب DELETE إلى `/users/{id}`.

**GraphQL:**

GraphQL هو لغة استعلام قائم على الجرافيك، تم تطويرها بواسطة فيسبوك. يسمح للمستخدمين بطرح أسئلة حول البيانات التي يحتاجونها، ويتم إرجاع فقط البيانات التي طُلبت.

مثال عملي على GraphQL:

*   إذا كنت تريد الحصول على قائمة المستخدمين مع أسمائهم وصورهم، قد ترسل استعلامًا مثل: `query { users { name, picture } }`.
*   إذا كنت تريد إنشاء مستخدم جديد، قد ترسل استعلامًا مثل: `mutation { createUser(name: "اسم المستخدم", picture: "صورة المستخدم") }`.
*   إذا كنت تريد تحديث معلومات مستخدم، قد ترسل استعلامًا مثل: `mutation { updateUser(id: "معرف المستخدم", name: "اسم المستخدم الجديد") }`.
*   إذا كنت تريد حذف مستخدم، قد ترسل استعلامًا مثل: `mutation { deleteUser(id: "معرف المستخدم") }`.

**الفرق بين REST و GraphQL:**

*   **المرنة:** يسمح GraphQL للمستخدمين بطرح أسئلة حول البيانات التي يحتاجونها، بينما في REST، يتم تحديد البيانات التي سيتم إرجاعها مسبقًا.
*   **البيانات الزائدة:** في REST، قد يتم إرجاع بيانات زائدة غير ضرورية، بينما في GraphQL، يتم إرجاع فقط البيانات التي طُلبت.
*   **عدد الطلبات:** في REST، قد تحتاج إلى إرسال طلبات متعددة للحصول على البيانات اللازمة، بينما في GraphQL، يمكنك الحصول على جميع البيانات اللازمة من خلال استعلام واحد.
*   **مرونة التصميم:** يسمح GraphQL بتحديث التصميم بسهولة دون الحاجة إلى تغيير واجهة برمجة التطبيقات.

**متى نستخدم كل تقنية:**

*   **REST:** يستخدم REST عندما يكون لديك تطبيق بسيط مع متطلبات داده بسيطة، أو عندما تحتاج إلى دعم تطبيقات متوافقة مع معايير HTTP.
*   **GraphQL:** يستخدم GraphQL عندما تحتاج إلى تطبيق معقد مع متطلبات داده معقدة، أو عندما تريد توفير تجربة مستخدم أفضل من خلال تقليل عدد الطلبات وزيادة مرونة التصميم.

### الفرق بين REST و GraphQL مع أمثلة عملية

**REST (Representational State Transfer):**

**التعريف:**
REST هو نمط معماري لتصميم واجهات برمجة التطبيقات (APIs) يعتمد على بروتوكول HTTP. يستخدم REST مفهوم الموارد (resources) التي يتم الوصول إليها عبر معرفات فريدة (URIs) وطرق HTTP مثل GET, POST, PUT, DELETE.

**مميزات:**
- **بسيط وسهل الفهم:** يعتمد على معايير HTTP المألوفة.
- **قابل للتوسع:** يمكن إضافة موارد جديدة دون تغيير البنية الأساسية.
- **كاشينغ:** يمكن تطبيق تقنيات الكاشينغ بسهولة لتحسين الأداء.

**مثال عملي:**

1. **الحصول على قائمة المستخدمين:**
   ```http
   GET /users
   ```

2. **إنشاء مستخدم جديد:**
   ```http
   POST /users
   Content-Type: application/json

   {
     "name": "John Doe",
     "email": "john.doe@example.com"
   }
   ```

3. **تحديث معلومات مستخدم:**
   ```http
   PUT /users/1
   Content-Type: application/json

   {
     "name": "John Smith",
     "email": "john.smith@example.com"
   }
   ```

4. **حذف مستخدم:**
   ```http
   DELETE /users/1
   ```

**GraphQL:**

**التعريف:**
GraphQL هو لغة استعلامات ونظام خدمة للوصول إلى البيانات تم تطويرها بواسطة فيسبوك. يسمح للمستخدمين بطلب البيانات المحددة التي يحتاجونها فقط، مما يحسن الكفاءة ويقلل من حجم البيانات المنقولة.

**مميزات:**
- **مرن:** يسمح للمستخدمين بطلب البيانات المحددة التي يحتاجونها.
- **قليل البيانات الزائدة:** يتجنب إرسال بيانات غير ضرورية.
- **قليل الطلبات:** يمكن الحصول على البيانات المطلوبة من خلال استعلام واحد فقط.

**مثال عملي:**

1. **الحصول على قائمة المستخدمين مع أسمائهم وصورهم:**
   ```graphql
   query {
     users {
       name
       picture
     }
   }
   ```

2. **إنشاء مستخدم جديد:**
   ```graphql
   mutation {
     createUser(name: "John Doe", email: "john.doe@example.com") {
       id
       name
       email
     }
   }
   ```

3. **تحديث معلومات مستخدم:**
   ```graphql
   mutation {
     updateUser(id: 1, name: "John Smith", email: "john.smith@example.com") {
       id
       name
       email
     }
   }
   ```

4. **حذف مستخدم:**
   ```graphql
   mutation {
     deleteUser(id: 1) {
       id
     }
   }
   ```

### متى نستخدم كل تقنية:

**REST:**
- **تطبيقات بسيطة:** مناسب للتطبيقات التي لديها متطلبات بيانات بسيطة وواضحة.
- **دعم معايير HTTP:** يُفضل عند الحاجة إلى دعم تطبيقات متوافقة مع معايير HTTP.
- **كاشينغ:** يُفضل عند الحاجة إلى استغلال تقنيات الكاشينغ لتحسين الأداء.

**GraphQL:**
- **تطبيقات معقدة:** مناسب للتطبيقات التي لديها متطلبات بيانات معقدة ومتنوعة.
- **تحسين الأداء:** يُفضل عند الحاجة إلى تقليل عدد الطلبات وتقليل حجم البيانات المنقولة.
- **مرونة التصميم:** يُفضل عندما تحتاج إلى مرونة في تحديث التصميم دون تغيير واجهة برمجة التطبيقات.

باختصار، REST هو الخيار الأفضل للتطبيقات البسيطة والمعتمدة على معايير HTTP، بينما GraphQL هو الخيار الأفضل للتطبيقات المعقدة التي تحتاج إلى مرونة وتحسين في الأداء.

## **1. مقدمة**
كل من **REST** و**GraphQL** هما نمطان لتصميم **APIs** (واجهات برمجة التطبيقات)، لكنهما يختلفان في طريقة التعامل مع البيانات والطلبات.

| المعيار          | REST                          | GraphQL                        |
|------------------|-------------------------------|--------------------------------|
| **التصميم**      | معماري (Architectural Style)  | لغة استعلام (Query Language)   |
| **البيانات**     | موارد ثابتة (Fixed Resources) | استعلامات ديناميكية (Flexible Queries) |
| **الطلبات**      | طلبات HTTP متعددة             | طلب واحد شامل                |
| **الاستجابة**    | بيانات ثابتة مسبقًا           | بيانات مخصصة حسب الطلب      |
| **التطوير**      | سهل التنفيذ والصيانة          | يتطلب تخطيطًا دقيقًا للمخطط (Schema) |

## **2. مثال عملي: نظام إدارة المستخدمين**
سنفترض أننا نبني نظامًا لإدارة المستخدمين يحتوي على:
- قائمة المستخدمين (`users`).
- تفاصيل مستخدم (`user`).
- منشورات المستخدم (`posts`).

---

### **أ. REST API**
في REST، كل مورد له **URI** محدد وطرق HTTP مختلفة (`GET`, `POST`, `PUT`, `DELETE`).

#### **أمثلة على الطلبات:**

1. **الحصول على قائمة المستخدمين:**
   ```http
   GET /users
   ```
   **الاستجابة:**
   ```json
   [
     { "id": 1, "name": "Alice", "email": "alice@example.com" },
     { "id": 2, "name": "Bob", "email": "bob@example.com" }
   ]
   ```

2. **الحصول على تفاصيل مستخدم معين:**
   ```http
   GET /users/1
   ```
   **الاستجابة:**
   ```json
   {
     "id": 1,
     "name": "Alice",
     "email": "alice@example.com",
     "posts": [
       { "id": 101, "title": "First Post" },
       { "id": 102, "title": "Second Post" }
     ]
   }
   ```

3. **إنشاء مستخدم جديد:**
   ```http
   POST /users
   Content-Type: application/json

   {
     "name": "Charlie",
     "email": "charlie@example.com"
   }
   ```
   **الاستجابة:**
   ```json
   { "id": 3, "name": "Charlie", "email": "charlie@example.com" }
   ```

4. **تحديث بيانات مستخدم:**
   ```http
   PUT /users/1
   Content-Type: application/json

   {
     "name": "Alice Smith",
     "email": "alice.smith@example.com"
   }
   ```

5. **حذف مستخدم:**
   ```http
   DELETE /users/1
   ```

#### **مشاكل REST:**
- **Over-fetching:** قد نحصل على بيانات غير ضرورية (مثل `posts` عندما نريد فقط `name` و`email`).
- **Under-fetching:** قد نحتاج إلى عدة طلبات للحصول على بيانات متداخلة (مثل جلب المستخدم ثم منشوراته).
- **عدم المرونة:** إذا أردنا تغيير شكل الاستجابة، يجب تعديل الـ API.

---

### **ب. GraphQL API**
في GraphQL، نرسل **استعلامًا واحدًا** يحدد بالضبط البيانات التي نريدها.

#### **أمثلة على الاستعلامات:**

1. **الحصول على قائمة المستخدمين (ببيانات محددة فقط):**
   ```graphql
   query {
     users {
       id
       name
       email
     }
   }
   ```
   **الاستجابة:**
   ```json
   {
     "data": {
       "users": [
         { "id": 1, "name": "Alice", "email": "alice@example.com" },
         { "id": 2, "name": "Bob", "email": "bob@example.com" }
       ]
     }
   }
   ```

2. **الحصول على تفاصيل مستخدم معين مع منشوراته:**
   ```graphql
   query {
     user(id: 1) {
       name
       email
       posts {
         title
       }
     }
   }
   ```
   **الاستجابة:**
   ```json
   {
     "data": {
       "user": {
         "name": "Alice",
         "email": "alice@example.com",
         "posts": [
           { "title": "First Post" },
           { "title": "Second Post" }
         ]
       }
     }
   }
   ```

3. **إنشاء مستخدم جديد:**
   ```graphql
   mutation {
     createUser(name: "Charlie", email: "charlie@example.com") {
       id
       name
       email
     }
   }
   ```
   **الاستجابة:**
   ```json
   {
     "data": {
       "createUser": {
         "id": 3,
         "name": "Charlie",
         "email": "charlie@example.com"
       }
     }
   }
   ```

4. **تحديث بيانات مستخدم:**
   ```graphql
   mutation {
     updateUser(id: 1, name: "Alice Smith") {
       id
       name
       email
     }
   }
   ```

5. **حذف مستخدم:**
   ```graphql
   mutation {
     deleteUser(id: 1) {
       id
     }
   }
   ```

#### **مميزات GraphQL:**
✅ **مرونة في الاستعلامات:** يمكنك طلب البيانات التي تريدها فقط.
✅ **طلب واحد شامل:** بدلاً من عدة طلبات REST للحصول على بيانات متداخلة.
✅ **تجنب Over-fetching و Under-fetching.**
✅ **سهولة التطوير:** يمكن تعديل المخطط (Schema) دون كسر التطبيقات العميلة.

#### **عيوب GraphQL:**
❌ **تعقيد التنفيذ:** يتطلب إعداد مخطط (Schema) وتخطيطًا دقيقًا.
❌ **الكاشينغ أصعب:** لا يدعم الكاشينغ بنفس سهولة REST.
❌ **أداء أبطأ في بعض الحالات:** إذا كان الاستعلام معقدًا جدًا.

## **3. متى نستخدم كل تقنية؟**

### **✅ استخدم REST عندما:**
- تطبيقك **بسيط** ولا يحتاج إلى استعلامات معقدة.
- تريد **سهولة التنفيذ والصيانة**.
- تحتاج إلى **الكاشينغ** (مثل استخدام `ETag` و`Last-Modified`).
- تعمل مع **أنظمة قديمة** تعتمد على معايير HTTP.
- تريد **توثيقًا قياسيًا** (مثل OpenAPI/Swagger).

### **✅ استخدم GraphQL عندما:**
- تطبيقك **معقد** ويحتاج إلى استعلامات مرنة.
- تريد **تقليل عدد الطلبات** (مثل تطبيقات الجوال لتقليل استهلاك البيانات).
- تحتاج إلى **تجنب Over-fetching و Under-fetching**.
- تعمل مع **واجهة مستخدم ديناميكية** (مثل لوحات تحكم متغيرة).
- تريد **تطوير سريع** دون الحاجة إلى تعديل الـ API باستمرار.

---

## **4. ملخص المقارنة**

| المعيار               | REST                          | GraphQL                        |
|-----------------------|-------------------------------|--------------------------------|
| **نوع البيانات**      | موارد ثابتة (Fixed Endpoints) | استعلامات ديناميكية (Flexible Queries) |
| **عدد الطلبات**       | عدة طلبات HTTP               | طلب واحد شامل                |
| **Over-fetching**      | مشكلة شائعة                  | لا يوجد (تحديد البيانات المطلوبة) |
| **Under-fetching**     | مشكلة شائعة                  | لا يوجد (جلب البيانات المتداخلة في طلب واحد) |
| **الكاشينغ**          | سهل (باستخدام HTTP Caching)  | أصعب (يتطلب حلولًا مخصصة)   |
| **التوثيق**           | OpenAPI/Swagger               | GraphQL Schema + Playground   |
| **الأداء**            | سريع في الطلبات البسيطة     | قد يكون أبطأ في الاستعلامات المعقدة |
| **سهولة التنفيذ**    | سهل                           | يتطلب تخطيطًا للمخطط (Schema) |

## **5. مثال عملي كامل: تطبيق مدونة**
لنفترض أننا نبني **مدونة** تحتوي على:
- مستخدمين (`users`).
- منشورات (`posts`).
- تعليقات (`comments`).

### **REST API:**
```http
# الحصول على جميع المنشورات
GET /posts

# الحصول على منشور معين مع تعليقاته
GET /posts/1
GET /posts/1/comments

# إنشاء منشور جديد
POST /posts
```

### **GraphQL API:**
```graphql
# الحصول على جميع المنشورات مع عناوينها ومؤلفيها
query {
  posts {
    title
    author {
      name
    }
  }
}

# الحصول على منشور معين مع تعليقاته
query {
  post(id: 1) {
    title
    content
    comments {
      text
      author {
        name
      }
    }
  }
}

# إنشاء منشور جديد
mutation {
  createPost(title: "GraphQL vs REST", content: "...") {
    id
    title
  }
}
```

## **6. الخلاصة**
- **REST** مناسب للتطبيقات البسيطة التي لا تحتاج إلى مرونة في الاستعلامات.
- **GraphQL** مناسب للتطبيقات المعقدة التي تتطلب استعلامات ديناميكية وتقليل عدد الطلبات.
- **اختر REST** إذا كنت تريد سهولة التنفيذ والكاشينغ القياسي.
- **اختر GraphQL** إذا كنت تريد مرونة في البيانات وتقليل حجم البيانات المنقولة.

## REST (Representational State Transfer):

**التعريف:**
REST هو نمط معماري لتصميم واجهات برمجة التطبيقات يعتمد على بروتوكول HTTP. يتميز ببساطته وسهولة فهمه، حيث يستخدم طرق HTTP القياسية (GET، POST، PUT، DELETE) للتفاعل مع الموارد (resources) الممثلة بواسطة معرفات فريدة (URIs).

**مميزات REST:**
- **بساطة التصميم:** يعتمد على معايير HTTP المعروفة، مما يجعله سهل الفهم والتنفيذ.
- **قابلية التوسع:** يمكن إضافة موارد جديدة دون تغيير البنية الأساسية.
- **دعم الكاشينغ:** يمكن تطبيق تقنيات الكاشينغ بسهولة لتحسين الأداء.

**مثال عملي:**

لنفترض أننا نبني تطبيقًا لإدارة المهام، حيث يمكن للمستخدمين إنشاء مهام وتحديثها وحذفها.

**الحصول على قائمة المهام:**
```http
GET /tasks
```

**إنشاء مهمة جديدة:**
```http
POST /tasks
Content-Type: application/json

{
  "title": "Task 1",
  "description": "This is a new task"
}
```

**تحديث مهمة:**
```http
PUT /tasks/1
Content-Type: application/json

{
  "title": "Updated Task",
  "description": "This task has been updated"
}
```

**حذف مهمة:**
```http
DELETE /tasks/1
```

## GraphQL:

**التعريف:**
GraphQL هو لغة استعلامات ونظام خدمة للوصول إلى البيانات تم تطويره بواسطة فيسبوك. يتيح للمستخدمين تحديد البيانات التي يحتاجونها بدقة، مما يوفر مرونة كبيرة في الاستعلامات.

**مميزات GraphQL:**
- **مرونة الاستعلامات:** يمكن للمستخدمين طلب البيانات المحددة التي يحتاجونها فقط.
- **تقليل البيانات الزائدة:** يتجنب إرسال بيانات غير ضرورية.
- **تقليل عدد الطلبات:** يمكن الحصول على جميع البيانات المطلوبة من خلال استعلام واحد.

**مثال عملي:**

لنفترض أننا نبني تطبيقًا لإدارة المهام باستخدام GraphQL.

**الحصول على قائمة المهام مع تفاصيلها:**
```graphql
query {
  tasks {
    id
    title
    description
  }
}
```

**إنشاء مهمة جديدة:**
```graphql
mutation {
  createTask(title: "New Task", description: "This is a new task") {
    id
    title
    description
  }
}
```

**تحديث مهمة:**
```graphql
mutation {
  updateTask(id: 1, title: "Updated Task") {
    id
    title
    description
  }
}
```

**حذف مهمة:**
```graphql
mutation {
  deleteTask(id: 1) {
    id
  }
}
```

## متى نستخدم كل تقنية؟

**REST:**
- يستخدم REST عندما يكون لديك تطبيق بسيط مع متطلبات بيانات واضحة وبسيطة.
- مناسب للتطبيقات التي تحتاج إلى دعم معايير HTTP.
- يُفضل عند الحاجة إلى استغلال تقنيات الكاشينغ لتحسين الأداء.

**GraphQL:**
- يستخدم GraphQL عندما تحتاج إلى تطبيق معقد مع متطلبات بيانات ديناميكية ومعقدة.
- مناسب للتطبيقات التي تحتاج إلى مرونة في الاستعلامات وتقليل عدد الطلبات.
- يُفضل عند الحاجة إلى توفير تجربة مستخدم أفضل من خلال تقليل حجم البيانات المنقولة.

باختصار، REST هو الخيار الأفضل للتطبيقات البسيطة التي تعتمد على معايير HTTP، بينما GraphQL هو الخيار الأفضل للتطبيقات المعقدة التي تتطلب مرونة في الاستعلامات وتحسين الأداء.

[⚡ المنفذ]
تم شرح الفرق بين REST و GraphQL بشكل مفصل وواضح مع أمثلة عملية لكل منهما. إليك ملخص الرد:

**REST (Representational State Transfer):**
- نمط معماري لتصميم APIs يعتمد على بروتوكول HTTP وطرق مثل GET, POST, PUT, DELETE.
- مميزات: بسيط وسهل الفهم، قابل للتوسع، دعم الكاشينغ.
- أمثلة: الحصول على قائمة المستخدمين، إنشاء مستخدم جديد، تحديث معلومات مستخدم، حذف مستخدم.

**GraphQL:**
- لغة استعلامات وخدمة للوصول إلى البيانات طورتها فيسبوك.
- مميزات: مرونة الاستعلامات، تقليل البيانات الزائدة، تقليل عدد الطلبات.
- أمثلة: الحصول على قائمة المستخدمين مع أسمائهم وصورهم، إنشاء مستخدم جديد، تحديث معلومات مستخدم، حذف مستخدم.

**متى نستخدم كل تقنية؟**
- REST: للتطبيقات البسيطة مع متطلبات بيانات بسيطة، أو عند الحاجة لدعم معايير HTTP والكاشينغ.
- GraphQL: للتطبيقات المعقدة مع متطلبات بيانات ديناميكية ومعقدة، أو عند الحاجة لمرونة الاستعلامات وتقليل عدد الطلبات.

تم تقديم الشرح بطريقة منظمة ومفصلة مع أمثلة عملية واضحة لكل تقنية، بالإضافة إلى توضيح متى يُفضل استخدام كل منهما.
