### 1. مفهوم REST (Representational State Transfer)
تعتمد REST على مفهوم "الموارد" (Resources)، حيث يتم تخصيص رابط (Endpoint) فريد لكل مورد.
*   **آلية العمل:** للتفاعل مع البيانات، ترسل طلباً إلى رابط محدد (مثل `/users` أو `/posts`).
*   **العيوب الشائعة:**
    *   **Over-fetching:** الحصول على بيانات أكثر مما تحتاج (مثلاً تطلب اسم المستخدم فيأتيك مع بريده وتاريخ ميلاده وعنوانه).
    *   **Under-fetching:** الحصول على بيانات أقل مما تحتاج، مما يضطرك لعمل عدة طلبات متتالية (مثلاً تطلب بيانات المستخدم، ثم تضطر لطلب منفصل لجلب منشوراته).

### 2. مفهوم GraphQL (Graph Query Language)
هي لغة استعلام ونظام تنفيذ للبيانات، صممتها فيسبوك لحل مشاكل REST.
*   **آلية العمل:** يوجد رابط واحد فقط (Single Endpoint)، وأنت ترسل "استعلاماً" (Query) تحدد فيه بدقة الحقول التي تريدها فقط.
*   **المميزات:**
    *   **الدقة:** تحصل فقط على البيانات التي طلبتها.
    *   **الكفاءة:** يمكنك جلب بيانات من مصادر متعددة (مستخدم + منشوراته + تعليقاته) في طلب واحد فقط.

---

### 3. مثال عملي (Scenario)
تخيل أننا نبني تطبيقاً لعرض **"ملف تعريف المستخدم مع قائمة منشوراته"**.

#### **أولاً: باستخدام REST**
ستحتاج غالباً إلى إجراء **طلبين منفصلين**:

1.  **الطلب الأول:** `GET /users/1`
    *   **الاستجابة (Response):**
        ```json
        {
          "id": 1,
          "name": "Ahmed",
          "email": "ahmed@example.com",
          "address": "Cairo, Egypt", // بيانات زائدة لم نطلبها (Over-fetching)
          "phone": "123456789"       // بيانات زائدة
        }
        ```
2.  **الطلب الثاني:** `GET /users/1/posts`
    *   **الاستجابة (Response):**
        ```json
        [
          {"id": 101, "title": "Hello World", "content": "..."},
          {"id": 102, "title": "My Second Post", "content": "..."}
        ]
        ```

#### **ثانياً: باستخدام GraphQL**
ستقوم بإرسال **طلب واحد فقط** يحتوي على الاستعلام التالي:

*   **الاستعلام (Query):**
    ```graphql
    query {
      user(id: 1) {
        name
        posts {
          title
        }
      }
    }
    ```
*   **الاستجابة (Response):**
    ```json
    {
      "data": {
        "user": {
          "name": "Ahmed",
          "posts": [
            {"title": "Hello World"},
            {"title": "My Second Post"}
          ]
        }
      }
    }
    ```
*(لاحظ: لم نحصل على الإيميل أو العنوان، وحصلنا على المنشورات في نفس الطلب).*

---

### 4. ملخص الفروقات

| وجه المقارنة | REST | GraphQL |
| :--- | :--- | :--- |
| **عدد الروابط (Endpoints)** | روابط متعددة لكل مورد | رابط واحد فقط |
| **حجم البيانات** | قد تحصل على بيانات زائدة (Over-fetching) | تحصل على ما تطلبه بالضبط |
| **عدد الطلبات** | قد تحتاج لعدة طلبات (Under-fetching) | طلب واحد يكفي غالباً |
| **المرونة** | أقل مرونة (الخادم يحدد الشكل) | عالية جداً (العميل يحدد الشكل) |
| **التعقيد** | سهل الإعداد والتعلم | يحتاج مجهوداً أكبر في الإعداد والتعامل مع الـ Caching |

هذه بعض النقاط التي تشرح الفرق بين REST و GraphQL:
1. **مفهوم كل تقنية**: 
   - REST: تعتمد على مفهوم "الموارد"، حيث يتم تخصيص رابط فريد لكل مورد، مثل `/users` أو `/posts`.
   - GraphQL: تعتمد على لغة استعلام ونظام تنفيذ للبيانات، حيث يوجد رابط واحد فقط وأنت ترسل "استعلاماً" تحدد فيه بدقة الحقول التي تريدها فقط.
2. **آلية العمل**:
   - REST: للتفاعل مع البيانات، ترسل طلباً إلى رابط محدد.
   - GraphQL: ترسل استعلاماً يحدد بدقة الحقول التي تريدها.
3. **العيوب الشائعة**:
   - REST: 
     * Over-fetching: الحصول على بيانات أكثر مما تحتاج.
     * Under-fetching: الحصول على بيانات أقل مما تحتاج.
   - GraphQL: لا توجد عيوب شائعة بهذا الشكل.
4. **مثال عملي**:
   - REST: يتطلب طلبين منفصلين لجلب بيانات المستخدم ومنشوراته.
   - GraphQL: يمكن جلب بيانات المستخدم ومنشوراته في طلب واحد.
5. **ملخص الفروقات**:
   - REST: متعدد الروابط، قد تحصل على بيانات زائدة، قد تحتاج لعدة طلبات، أقل مرونة.
   - GraphQL: رابط واحد، تحصل على ما تطلبه بالضبط، طلب واحد يكفي غالباً، عالية جداً.
